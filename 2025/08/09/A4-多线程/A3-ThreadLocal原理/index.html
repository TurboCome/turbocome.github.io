发表于 2025-08-09 分类于 多线程 阅读次数：

ThreadLocal原理

主要作用

线程间的数据隔离，别的线程使用get() 方法没办法拿到其他线程的值。每个线程 Thread 都维护自己的threadLocals变量，在每个线程创建 ThreadLocal 时，实际上数据是存在 线程Thread的 threadLocals中，其他线程没办法拿到，从而实现数据隔离。

value值存在哪里？

threadLocal的value值是存储在ThreadLocalMap中，ThreadLocalMap 是ThreadLocal的一个静态内部类，它的 key是 ThreadLocal实例对象，value是一个Object对象。每个线程实例对应一个 TheadLocalMap实例，在同一个线程里实例化很多 ThreadLocal来存储很多种类型的值，这些ThreadLocal实例分别作为key，对应各自的value，存储在 Entry table 数组中。

set获取当前线程的ThreadLocalMap，然后往map里添加 KV，K是当前ThreadLocal实例，V是我们传入的value

get获取当前线程对应的私有变量，是之前set或通过 initialValue的值；

线程的一些局部变量和引用：Stack（栈）区； 普通对象：Heap（堆）区

1.线程运行时，定义的 TheadLocal对象被初始化，存储在Heap，同时线程运行的栈区保存指向该实例的引用，也就是图中的ThreadLocalRef；

2.当 ThreadLocal的 set / get 被调用时，虚拟机会根据当前线程引用，查看对应的TheadLocalMap实例是否被创建，如果没有则创建并初始化；

3.虚线，表示 key对应的 ThreadLocal实例的引用是个弱引用；

使用场景

1.需要存储线程私有变量时；

2.需要实现线程安全的变量时；

3.需要减少线程资源竞争时；

ThreadLocalMap解决Hash冲突

注意：没有链表、红黑树

一个线程可以有多个 TreadLocal 来存放不同类型的对象，他们都放到当前线程的 ThreadLocalMap里，一个线程由一个数组来存储。ThreadLocalMap存储时，给每一个ThreadLocal对象一个 threadLocalHashCode，插入过程中根据 ThreadLocal对象的hash值，定位到 table中的位置 id ，然后判断当前位置 是否为空，为空就初始化一个Entry对象放在位置id上；线性探测解决冲突，步长+1,-1

如果位置id 不空，这个Entry对象的 key = 计算出来的hash值，则更新 Entry中的value；

如果位置id 不空，且 Entry 对象的 key !=计算出来的hash值， 则找下一个空位置；

问题：set和get如果冲突严重的话，效率会降低。

ThreadLocal内存泄漏

ThreadLocalMap 使用的 key 为 ThreadLocal 的弱引用，而 value 是强引用。

问题：如果ThreadLocal 没有被外部强引用，在垃圾回收时 key 会被清理掉，而 value不会被清理掉。这样 ThreadLocalMap中会出现 key为null的Entry。假如不做任何措施，value 永远无法被GC 回收，导致内存泄露。

解决方式：每次使用完 ThreadLocal，调用 remove()方法，将Entry节点和Map的引用关系移除，这样整个Entry对象在GC Roots分析后就变成不可达，下次GC的时候就可以被回收。

ThreadLocalMap的key为什么使用弱引用？

如果key 使用强引用：

引用的 ThreadLocal的对象被回收，但是ThreadLocalMap还持有ThreadLocal的强引用，如果没有手动删除，ThreadLocal不会被回收，导致Entry内存泄漏。

key 使用弱引用:

引用的ThreadLocal的对象被回收，由于ThreadLocalMap持有ThreadLocal的弱引用，即使没有手动删除，ThreadLocal 也会被回收。value在下一次ThreadLocalMap调用set、get、remove时会被清除。

比较两种情况发现：由于ThreadLocalMap的生命周期跟Thread一样长，如果都没有手动删除对应key，都会导致内存泄漏，但是使用弱引用可以多一层保障：弱引用ThreadLocal不会内存泄漏，对应的 value在下一次ThreadLocalMap调用 remove的时候会被清除。

ThreadLocalMap使用ThreadLocal的弱引用作为key，如果一个ThreadLocal没有了外部强引用（前提是这个ThreadLocal引用是定义在线程中的局部变量或在请求后被赋值为null，而不是一个final static静态变量字段） 来引用它，那么系统 GC 的时候，这个ThreadLocal势必会被回收，ThreadLocalMap中就会出现key为null的Entry ，没有办法访问这些key为null的Entry的value ，如果当前线程一直不结束（线程池中的核心线程） ，这些key为null的Entry的value就会一直存在一条强引用 链：Thread引用 -> Thread -> ThreadLocalMap -> Entry -> value 永远无法回收，造成内存泄漏 。

脏数据问题

在线程池中的核心线程会实现线程的复用 ，当前线程对象执行了上一次任务 后没有调用ThreadLocal对象的remove()方法 清除数据，仍使用该线程对象执行第二次任务的时没有执行该ThreadLocal对象的set()方法 ，而是直接使用get()方法 会得到上一次任务的脏数据 。
