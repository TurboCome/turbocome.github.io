AQS原理

发表于 2025-08-10 分类于 多线程 阅读次数：

AbstractQueuedSynchronizer（AQS）

应用场景：ReentrantLock、CountDownLatch 都是基于AQS实现

AQS实现原理

  • AQS维护了一个volatile int state（共享变量）；
  • 一个 FIFO线程等待队列（多线程争用资源被阻塞时会进入此队列）；

state共享变量

由于是多线程共享变量，所以定义成 volatile，保证 state可见性，并通过 CAS来保证其并发修改的安全性。 初始化为0，多线程条件下线程要执行临界区代码，首先获取 state。 某个线程获取成功后 state +1，其他线程如果再想获取，将被阻塞到 FIFO等待队列中，等占有 state线程执行完，释放资源( state -1)后，会唤醒 FIFO中下一个等待的线程去获取 state。

FIFO双向链表队列

等待队列是一个双向链表，head 结点代表当前占用的线程，其他节点由于暂时获取不到锁，依次 排队等待锁的释放。

AQS资源共享方式

1.Exclusive（独占）：只有一个线程执行，如ReentrantLock，可分为公平锁和非公平锁；

  • 公平锁：按照线程在队列中的排队顺序，先到者先得锁；
  • 非公平锁：获取锁时，无视队列顺序直接抢锁，谁抢到是谁的；

2.Share（共享）：多线程同时执行，如CountDownLatch；

AQS内部维护一个线程的队列，队列由内部的节点组成。队列的节点分为SHARED和EXCLUSIVE分别为共享模式的节点和独占模式的节点。

节点的等待状态为waitStatus：

  • CANCELLED（1）：取消状态，当线程不再希望获取锁时，设置为取消状态；
  • SIGNAL（-1）：当前节点的后继者处于等待状态，当前节点的线程如果释放或取消了同步状态，通知后继节点；
  • CONDITION（-2）：等待队列的等待状态，当调用signal()时，进入同步队列；
  • PROPAGATE（-3）：共享模式，同步状态的获取的可传播状态
  • 0：初始状态

AQS内部结构

为了维护等待在条件变量上的等待线程，AQS维护一个条件变量等待队列，由 Condition.await() 引起阻塞的线程。一个重入锁可以生成 多个条件变量对象，因此一个重入锁 可能有多个条件变量等待队列。每个条件变量对象内部都维护一个等待列表。

同步等待队列，条件变量等待队列都使用 同一个 Node类作为链表的节点；Node 中包括链表的上一个元素 prev，下一个元素next 和线程对象thread，对于条件变量等待队列，还使用nextWaiter表示下一个等待在条件变量队列中的节点。

AQS加锁&释放锁详细过程

三个线程(线程1，2，3)同时来加锁/释放锁；

1.线程1加锁成功

AQS的state = 1，2/3线程按照顺序放入等待队列FIFO；

2.线程2/3加锁失败

线程1将state修改为1，线程2通过 CAS修改state值不会成功，加锁失败。

线程2执行 tryAcquire()后会返回false，接着执行addWaiter(Node.EXCLUSIVE)逻辑，将自己加入到一个FIFO等待队列中。

等待队列中的节点Node是一个双向链表；

3.线程1释放锁&线程2加锁

线程1释放锁后会 唤醒head节点的后置节点，就是线程2。线程2设置为head节点，然后置空之前的head节点数据，被置空的节点数据会被垃圾回收。

4.Contition实现原理

await()方法首先调用 addConditionWaiter()将当前线程加入到Condition队列。这里会用当前线程创建一个 Node节点，waitStatus为CONDITION。 接着会释放该节点的锁，调用之前解析过的release()方法，释放锁之后，会唤醒被挂起的线程2，线程2会继续尝试获取锁。

接着判断当前节点是否为 Condition队列中的头部节点，如果是则挂起 Condition中当前线程。此时线程1被挂起，线程2获取锁成功。线程2执行signal()方法，如果线程2已经获取到锁，此时AQS等待队列中已经没有数据。

Condition队列中只有线程1 创建的一个Node节点，且waitStatue为CONDITION，先通过CAS修改当前节点waitStatus为0，然后执行enq()方法将当前线程加入到等待队列中，并返回当前线程的前置节点，接着开始通过CAS修改当前节点的前置节点waitStatus为SIGNAL，并且唤醒当前线程。等线程二释放锁后，线程一会继续重试获取锁，结束。此时等待队列中数据如下图：
